Planificación y Organización

Planificación

Para la planificación del proyecto, inicialmente realizamos una lluvia de ideas sobre posibles juegos a desarrollar. Lo primero obviamente fueron los HTML y los CSS.

Puente de cristal

El jugador tiene a su disposición un número limitado de personas que deben cruzar un puente de dos baldosas.

Una de las baldosas es falsa y se romperá si el jugador la elige.

El objetivo es avanzar lo más lejos posible por el puente sin perder a todos los personajes.

Duelo de vaqueros

El jugador se enfrenta a la máquina en un duelo al estilo del viejo oeste.

Para iniciar el juego, el jugador debe mantener el clic hasta que se indique el momento de disparar.

Si el jugador tarda más que la máquina en soltar el clic, o lo suelta antes de tiempo, perderá.

El tiempo se genera aleatoriamente (mín. 8 seg, máx. 15 seg).

Se consideró agregar elementos distractores en la pantalla para aumentar la dificultad.

Tras la creación de los bocetos iniciales, comenzamos a desarrollar las páginas correspondientes.

Estructura

La estructura del código ha sido organizada de forma clara y accesible. En la carpeta principal se encuentran las páginas fundamentales (index.html, juego.html y documentacion.html), junto con las carpetas para los archivos CSS, JavaScript y recursos como imágenes.

Cada juego dispone de su propio archivo .css y .js, además de sus respectivas páginas de juego, victoria y derrota (e imágenes, si las tuviese).

Al final, pensamos que sería una buena idea tener una página llamada games.html, donde, mediante un iframe, se cargarían los archivos .html de los juegos que hiciésemos. Hacer esto nos simplificó la tarea de crear pantallas de victoria y derrota.

Descripción de los juegos

Nuestra página cuenta con tres juegos:

Quiz

El jugador responde preguntas de diversas categorías o de un tema específico dentro de un tiempo limitado.

Cada respuesta correcta suma puntos, mientras que una respuesta incorrecta resta puntos al puntaje final.

Tríos

El jugador tiene un contador de tiempo para encontrar todos los tríos de cartas.

Si selecciona incorrectamente, perderá tiempo; si acierta, ganará tiempo.

El objetivo es encontrar todos los tríos antes de que se agote el tiempo.

Guerrillas

El juego enfrenta a la máquina y al jugador en un tablero de tres columnas.

Cada jugador tiene una cantidad limitada de vida y debe colocar soldados en las columnas para atacar al rival.

Si un jugador no coloca soldados en una columna para defenderse, recibirá daño.

Los soldados tienen diferentes estadísticas (vida y daño), y los enfrentamientos entre ellos determinarán cuál se queda en la columna.

El juego finaliza cuando uno de los jugadores pierde toda su vida.

Estructura del Código

Estructurar el código ha sido una tarea pesada, dado la gran cantidad de archivos que hemos tenido que manejar. Dado que no hay mucho que comentar sobre estilos CSS y codigo HTML, creo lo mas importante a mencionar serán los archivos JavaScript.

Este archivo JavaScript tiene como objetivo actualizar el atributo src del iframe en la página games.html, lo que permite cargar un juego diferente basado en la URL de la página.

La función updateSrc() obtiene el valor del parámetro game de la URL utilizando URLSearchParams. Si el parámetro está presente, el iframe con el ID game-iframe tendrá su atributo src actualizado con la URL del juego especificado. Si no se encuentra el parámetro game en la URL, se muestra un mensaje de error en la consola.

Este proceso ocurre automáticamente cuando la página se carga, ya que la función updateSrc() está vinculada al evento load del objeto window.

Este es un temporizador básico que los juegos que trabajen con temporizadores usarán.

La función decrementTime() se ejecuta cada segundo mediante setInterval(). Esta función disminuye el valor de la variable time y actualiza el contenido del elemento <div id="timerDisplay"> para mostrar el tiempo restante.

Cuando el tiempo llega a 0, se ejecuta la función end(), que detiene el temporizador usando clearInterval(interval). Dentro de los juegos se deberá sustuir esta función para poner el codigo deseado al acabarse el tiempo.

Se define una variable questions que contiene un array de objetos, cada uno representando una pregunta con sus opciones y la respuesta correcta.

El juego muestra una pregunta a la vez mediante la función showQuestion(), que también crea dinámicamente los botones de respuesta. Cada respuesta tiene un color distintivo y un evento click para verificar si es correcta.

Si la respuesta es correcta, se suma un punto al marcador. Cuando el tiempo se acaba o el usuario responde, se llama a markAnswers() para mostrar los resultados de la pregunta actual.

El botón de "Siguiente" permite pasar a la siguiente pregunta. Cuando se responden todas, la función endTime() guarda la puntuación en localStorage y redirige a una página de resultados (quiz-end.html).

El juego también incluye la función shuffleArray() que mezcla las preguntas de forma aleatoria al comenzar.

Al iniciar el juego, se mezclan las cartas con la función shuffleCards(). Las cartas se representan con imágenes, y todas se colocan inicialmente con la parte trasera visible (una imagen por defecto que lo simula). Cada carta tiene un número como clase que identifica el ID de su tipo de carta.

Cuando el jugador hace clic en una carta, se ejecuta chooseCard(). Esta función gira la carta y la añade al array de cartas elegidas. Cuando se eligen tres cartas, se comparan los IDs dentro del array:

  • Si las tres cartas son iguales, se marcan como encontradas y se suma tiempo al temporizador.
  • Si son diferentes, se penaliza al jugador restando tiempo y se vuelven a girar las cartas después de un segundo.

No se permiten elegir cartas ya elegidas y encontradas, para ello se diferencian dichas cartas del resto con las clases .choosen y .found.

El progreso del jugador se indica con efectos visuales, como cambiar el color y el texto del elemento con ID penalization para mostrar penalizaciones o bonificaciones de tiempo.

El juego termina con dos condiciones:

  • Derrota: cuando se agota el tiempo, se cambia el contenido del iframe a trio-lose.html.
  • Victoria: cuando el jugador encuentra todos los tríos, se redirige a trio-win.html.

Al cargar el juego, se ejecuta la función initializeGame(), que se encarga de configurar todo el juego. Esta función llama a otras como updateShopWarriors() (que actualiza los guerreros en la tienda) y updateInfo() (que muestra la vida y el dinero del jugador y enemigo).

Las estadisticas de los soldados se guardan dentro de un array, al igual que las casillas del enemigo y del jugador.

El jugador puede seleccionar guerreros de la tienda, arrastrándolos a las celdas disponibles en su campo de batalla. Las celdas donde se pueden colocar guerreros se gestionan con las funciones addDragStartEvent(), onDrop(), y enableDragAndDrop(), que permiten realizar el arrastre y el "drop" (soltar) de los guerreros.

Cuando el jugador pasa el turno, la función nextTurn() se activa. Esta función realiza el combate entre los guerreros usando la función combat(), donde se calculan los daños entre guerreros comparando ambos arrays de casillas (la casilla enemiga 1 con la casilla jugador 1 y asi...). Después, el turno pasa al enemigo, que hace su movimiento con la función enemyIA(), la cual coloca guerreros de manera aleatoria en el campo de batalla.

Las funciones updateField() y updateInfo() se encargan de actualizar los valores de vida y las posiciones de los guerreros después de cada turno, asegurando que se muestren correctamente en la interfaz. La función checkEnd() verifica si alguno de los dos bandos ha ganado o perdido.

El jugador tiene un control total sobre el despliegue de guerreros, el combate y la administración de recursos como las monedas. La tienda y las celdas se actualizan en función de las decisiones del jugador, con una lógica que impide colocar guerreros si no se tiene suficiente dinero o si no hay espacio.

Problemas encontrados

Las respuestas correctas o incorrectas no se visualizaban después de que el jugador las seleccionara, lo que dificultaba la retroalimentación inmediata.

Solución:

La función que se usaba antes para validar las respuestas se había planteado mal desde el inicio y se llamaba cuando no era correcto. Hubo que hacerla de 0 en markAnswers() con un algoritmo mejor y una lógica más desarrollada.

El botón destinado a pasar a la siguiente pregunta en el quiz no estaba funcionando correctamente, lo que impedía avanzar por el cuestionario.

Solución:

Se mostraba constantemente, aun cuando no se habia elegido una respuesta todavía. Hubo que modificar el Javascript para que lo mostrase y ocultase cuando fuese necesario, además de cumplir su función com debería.

Una vez que el jugador llegaba a la pantalla final, no había información sobre si había ganado o perdido el quiz, lo que restaba claridad en la experiencia del usuario.

Solución:

No se estaba utilizando correctamente el localStorage. Con unos pequeños ajustes se corrigio para que pasase correctamente la puntuación y las preguntas totales.

Se pueden elegir cartas ya encontradas o cartas que se han elegido, lo que hacia el juego muy confuso.

Solución:

Se usaron las clases .found y .choosen para diferenciar las cartas del, además de estilos CSS que las diferencien visualmente y un control de estas en la función chooseCard().
